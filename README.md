## `0x04` рҹ“қ Spec-Driven Execution

```
 в–‘в–’в–“в–Ҳ Batch distillation is spec-driven вҖ” plan first, execute second. в–Ҳв–“в–’в–‘
```

Before any file is read or any agent is spawned, the orchestrator generates a **distillation spec** (`.distill-spec.md`) вҖ” a full execution plan that serves as review gate, execution tracker, and audit trail.

```
 в”Ңв”Җ SPEC-DRIVEN PIPELINE в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ
 в”Ӯ                                                                    в”Ӯ
 в”Ӯ  1. Scan         в”Җв–¶ List files + sizes (metadata only)               в”Ӯ
 в”Ӯ  2. Spec         в”Җв–¶ Write .distill-spec.md (execution plan)          в”Ӯ
 в”Ӯ  в”Җв”Җв”Җ REVIEW GATE в”Җв”Җв”Җ user confirms before proceeding в”Җв”Җв”Җ          в”Ӯ
 в”Ӯ  3. Pre-read     в”Җв–¶ Read all files via read tool (CJK-safe)          в”Ӯ
 в”Ӯ  4. Refine       в”Җв–¶ Adjust spec based on actual content              в”Ӯ
 в”Ӯ  5. Embed        в”Җв–¶ Inline content into task packets                 в”Ӯ
 в”Ӯ  6. Execute      в”Җв–¶ Parallel extractors (вҸі/вң…/вқҢ tracked in spec)    в”Ӯ
 в”Ӯ  7. Synthesize   в”Җв–¶ Merge, deduplicate, validate                    в”Ӯ
 в”Ӯ  8. Density      в”Җв–¶ Dynamic fission for oversized files              в”Ӯ
 в”Ӯ  9. Validate     в”Җв–¶ Run checklist                                   в”Ӯ
 в”Ӯ  10. Finalize    в”Җв–¶ Append results to spec                          в”Ӯ
 в”Ӯ                                                                    в”Ӯ
 в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ
```

### Why Spec-Driven?

| Advantage | How |
|---|---|
| **Review gate** | User sees the full plan before committing tokens/time. Adjust scope, dimensions, or file assignments before execution. |
| **Execution tracking** | Each dimensionвҖҷs status is updated in the spec as extractors run (вҸі dispatched вҶ’ вң… complete вҶ’ вқҢ failed). |
| **Audit trail** | After completion, the spec documents every decision: which files went to which extractor, what was skipped, and why. |
| **Reproducibility** | Re-running with the same spec produces the same file assignments. |
| **Debugging** | When an extractor produces poor output, the spec shows exactly what content it received. |
| **No wasted reads** | Orchestrator pre-reads all files once, embeds content in task packets. Extractors never touch disk вҖ” avoids CJK encoding issues and redundant I/O. |

### Pre-Execution Protocol

Before any distillation starts, the AI always:

1. **Clarifies requirements** вҖ” Who is the target? WhatвҖҷs the source? Which dimensions? What does the LLM already know (skip common knowledge)?
2. **Determines mode** вҖ” Interactive, Batch, or Hybrid.
3. **Presents the plan** вҖ” For batch: the `.distill-spec.md`. For interactive: interview outline.
4. **Gets explicit confirmation** вҖ” Never starts extraction until the user says go.

> *No surprise token burns. No "oops, it read 200 files I didnвҖҷt want." You see the plan, you approve the plan, then it runs.*
